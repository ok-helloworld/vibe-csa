# CoT 四步推理（逻辑漏洞）

逻辑漏洞不同于传统注入漏洞——它们没有明显的危险函数调用（Sink）。检测逻辑漏洞需要模拟攻击者思维，理解业务意图与实际实现之间的差距。

## 四步推理框架

### Step 1: 场景识别（Scene & Entry Recognition）

识别目标接口的业务场景和入口点：
- 这个接口做什么业务操作？
- 输入参数有哪些？分别代表什么？
- 正常业务流程是什么？
- 可能的滥用方式有哪些？

**触发条件**：T1 入口标记为以下场景之一时，必须执行 CoT 推理：
- FINANCIAL_TRANSACTION（支付、转账、退款、充值）
- PRIVILEGED_OPERATION（权限变更、角色分配、用户管理）
- RESOURCE_ALLOCATION（库存扣减、配额分配、资源创建）
- STATE_TRANSITION（订单状态变更、审批流程）

### Step 2: 防御机制审计（Defense Mechanism Audit）

审计现有的"锁"（Locks）和"盾"（Shields）：

**锁（Locks）**——访问控制层：
- 权限检查：是否需要特定角色/权限？
- 所有权验证：是否验证操作者与资源的所有关系？
- 状态前置条件：操作前是否验证资源状态？
- 并发控制：是否有锁/事务防止竞态条件？

**盾（Shields）**——数据验证层：
- 输入验证：参数类型、范围、格式是否校验？
- 业务规则检查：金额是否为正、数量是否充足？
- 异常处理：异常情况是否正确处理而非泄露信息？
- 幂等性保护：重复请求是否安全？

输出格式：
```
| 防护类型 | 具体机制 | 位置（file:line） | 强度 |
|----------|----------|-------------------|------|
| 权限检查 | @PreAuthorize("hasRole('ADMIN')") | XxxController.java:23 | 强 |
| 所有权验证 | 无 | — | 无 |
| 输入验证 | 金额 > 0 | XxxService.java:45 | 弱 |
```

### Step 3: 对抗沙盒模拟（Adversarial Sandbox Simulation）

设计 PoC 绕过方案：
- 如果缺少所有权验证，能否操作他人资源？（水平越权）
- 如果缺少状态前置条件，能否跳过必要步骤？（状态机绕过）
- 如果缺少并发控制，能否利用竞态条件？（并发攻击）
- 如果输入验证不完整，能否注入恶意数据？（数据篡改）

PoC 设计模板：
```
1. 构造请求：{HTTP Method} {URL}
2. 参数篡改：将 {参数名} 从 {正常值} 改为 {攻击值}
3. 预期结果：绕过 {缺失的防护}，实现 {攻击目标}
4. 实际结果：需要代码执行或人工验证确认
```

### Step 4: 漏洞结果判定（Vulnerability Result Judgment）

| 条件 | 判定 |
|------|------|
| 缺失关键锁/盾 + PoC 路径清晰 + 影响可确认 | CONFIRMED |
| 缺失防护但 PoC 路径不确定 | HYPOTHESIS |
| 有充分防护覆盖 | 不报告 |

## 推理模板

### 模板 1: 支付篡改

```
场景：订单支付接口
锁检查：
  - 金额是否从服务器端计算？（vs 客户端传入）
  - 订单状态是否验证？（防止重复支付/已退款订单支付）
  - 用户身份是否与订单匹配？
盾检查：
  - 金额范围验证（防止负数/溢出）
  - 货币类型验证
  - 折扣/优惠券逻辑是否可绕过
对抗模拟：
  - 篡改支付金额为 0.01
  - 篡改订单 ID 为他人订单
  - 并发发送两次支付请求
```

### 模板 2: 水平越权

```
场景：资源操作接口（查看/修改/删除）
锁检查：
  - 是否验证当前用户与资源所有者的关系？
  - ID 参数是否可被篡改为其他用户资源？
盾检查：
  - 资源 ID 是否从 session 获取（vs 从请求参数）
  - 是否有全局拦截器验证所有权
对抗模拟：
  - 将资源 ID 改为其他用户的 ID
  - 使用自己的认证 token 操作他人资源
```

### 模板 3: 库存并发

```
场景：库存扣减接口
锁检查：
  - 是否有数据库行锁/乐观锁？
  - 是否先查后改（存在 TOCTOU 窗口）？
盾检查：
  - 库存扣减是否为原子操作？
  - 是否有唯一约束防止超额扣减
对抗模拟：
  - 并发发送 N 个扣减请求
  - 利用 TOCTOU 窗口超额扣减
```

### 模板 4: 状态机绕过

```
场景：订单/审批状态变更接口
锁检查：
  - 状态变更是否验证前置状态？
  - 是否允许跳过必要中间状态？
盾检查：
  - 状态机是否有枚举约束？
  - 是否有状态变更日志
对抗模拟：
  - 直接将状态从 "pending" 改为 "completed"
  - 绕过 "approved" 中间状态
```

### 模板 5: CSRF

```
场景：状态变更接口
锁检查：
  - 是否有 CSRF Token？
  - Token 是否绑定 session 且不可预测？
盾检查：
  - 是否检查 Origin/Referer Header？
  - 是否使用 SameSite Cookie 属性
对抗模拟：
  - 构造恶意 HTML 页面，包含自动提交表单
  - 诱导已登录用户访问该页面
```

### 模板 6: Webhook 伪造

```
场景：接收第三方回调的 Webhook 接口（支付回调、消息推送、CI/CD 触发等）
锁检查：
  - 是否验证请求来源 IP/域名？
  - 是否验证签名/HMAC？（如 X-Hub-Signature、X-Signature）
  - 签名密钥是否足够随机（非硬编码/可猜测）？
  - 是否有重放保护（时间戳 + nonce）？
盾检查：
  - 回调中的金额/状态字段是否直接信任（而非从服务端查询）？
  - 同一事件 ID 是否可重复触发（幂等性缺失）？
  - 异常回调是否被静默处理（如 200 OK 但不执行业务）？
对抗模拟：
  - 伪造支付成功回调，修改 amount=0.01 或 status=SUCCESS
  - 重放合法的历史回调（若无 nonce）
  - 在签名验证前篡改 body（如果签名只验证部分字段）
  - 绕过 IP 白名单（若未验证 X-Forwarded-For 或 proxy header）
```

## 何时跳过 CoT

以下情况可以跳过 CoT 推理：
- 接口为纯只读操作（GET 查询，无状态变更）
- 接口为 PUBLIC_ACCESS 场景且无敏感数据
- 接口已被其他防护层（如网关、WAF）完全覆盖（需验证）
