# 覆盖率门禁

## 目的

防止 LLM 跳过"看起来不重要"的文件。入口点文件（T1）必须 100% 覆盖，无例外。

## 覆盖率计算

```
Tier 覆盖率 = 该 Tier 已审查文件数 / 该 Tier 总文件数 × 100%
总体覆盖率 = 已审查文件数 / 总文件数（不含 SKIP）× 100%
```

**已审查文件的严格定义**：满足以下全部条件才算已审查——

- **条件 A**：`findings-raw.md` 中存在该文件路径对应的 `## [REVIEWED]` 摘要块（无论该文件是否有漏洞发现）。仅在漏洞条目中被"顺带引用"的文件路径**不算满足条件 A**。
- **条件 B**：`## [REVIEWED]` 摘要块格式完整，包含以下所有子字段：
  ```
  ## [REVIEWED] path/to/File.java
  - Track: Sink驱动 / 控制驱动 / 双轨
  - Sinks found: <数量>（0 也必须显式写出）
  - Controls checked: <已检查的安全控制列表>
  - Summary: <1-2句说明结论>（如"未发现危险 Sink，认证中间件完整覆盖所有路由"）
  ```
- **条件 C**：T1 文件额外要求至少一条控制驱动分析记录（即使无漏洞）

**关键区分**：`## [REVIEWED] FileA.java`（摘要块）≠ `FileA.java:87 存在 SQL 注入`（漏洞引用）。前者是覆盖率凭证，后者不是。无漏洞的文件必须有摘要块才计入已审查，这是防止 LLM 只扫"有趣的文件"跳过干净文件的核心机制。

## 阈值

| 项目规模 | EALOC | T1 | T2 | T3 | 总体 |
|---------|-------|----|----|----|------|
| 小型 | <15,000 | 100% | 95% | 85% | 100% |
| 中型 | 15,000-50,000 | 100% | 95% | 85% | 95% |
| 大型 | >50,000 | 100% | 90% | 80% | 90% |

## 门禁规则

1. **T1 必须 100%**：无论项目规模，T1 文件（入口点）必须逐一审查。这是硬性要求。
2. 四个条件**全部满足**才能通过门禁。
3. T1 不达标时，**无论其他指标如何**，门禁直接失败。
