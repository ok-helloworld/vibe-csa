# 评分系统

vibe-csa 使用双评分系统：DKTSS 用于漏洞优先级排序，3D 公式用于严重程度分级。两者互补，同时出现在最终报告中。

## DKTSS 优先级评分

```
Score = min(10, Base - Friction + Weapon + Ver)
```

### Base（基础分）

| 漏洞类型 | Base 分 | 说明 |
|----------|---------|------|
| RCE（命令执行、反序列化、SSTI） | 10 | 直接代码执行 |
| SQL 注入（数据泄露/篡改） | 6-8 | 取决于数据库权限 |
| SSRF（内网访问） | 4-7 | 取决于网络可达性 |
| 文件上传（可执行） | 8-9 | 取决于上传路径和验证 |
| 文件读取（路径遍历） | 5-7 | 取决于可读文件范围 |
| XXE | 6-8 | 取决于解析器配置 |
| 认证绕过 | 6-8 | 取决于绕过后权限 |
| CSRF | 4-6 | 取决于操作敏感性 |
| XSS（存储型） | 5-7 | 取决于上下文 |
| XSS（反射型） | 3-5 | 取决于交互需求 |
| 弱加密/硬编码密钥 | 3-5 | 取决于数据敏感性 |
| 信息泄露 | 2-4 | 取决于泄露内容 |

### Friction（利用摩擦度）

分三个维度独立计算，**防止双重扣分**：访问路径和权限阈值描述同一维度的两个视角，取绝对值较大的一项，再累加交互复杂度：

```
Friction = max(|访问路径|, |权限阈值|) + |交互复杂度|
```

**访问路径**

| 条件 | 扣分 |
|------|------|
| 公开可达（无需认证） | 0 |
| 需要普通用户认证 | -1 |
| 需要管理员/特权认证 | -3 |
| 需要内网访问 | -2 |

**权限阈值**

| 条件 | 扣分 |
|------|------|
| 无需权限 | 0 |
| 任意注册用户 | -1 |
| 特定角色/组 | -2 |
| 仅管理员/超级用户 | -3 |

**交互复杂度**

| 条件 | 扣分 |
|------|------|
| 自动触发（无需交互） | 0 |
| 需要用户点击链接 | -1 |
| 需要多步时序攻击 | -3 |

> **示例**：漏洞需要管理员认证（访问路径 -3）且仅管理员角色（权限阈值 -3）→ `max(3, 3) = 3`，不是 6。

### Weapon（武器化程度）

| 级别 | 加分 |
|------|------|
| 成熟 EXP 工具公开可用 | +1 |
| 有 PoC 但未武器化 | 0 |
| 理论可行，无公开 PoC | -2 |

### Ver（版本因素）

| 情况 | 加分 |
|------|------|
| 当前版本，无补丁 | 0 |
| 新版本已修复，但目标仍为旧版 | +1 |
| 仅影响已弃用版本 | -1 |

### DKTSS 等级

| 分数 | 等级 | 响应 |
|------|------|------|
| 9-10 | Critical | 立即修复，24小时内 |
| 7-8 | High | 优先修复，1周内 |
| 5-6 | Medium | 计划修复，1月内 |
| 3-4 | Low | 视情况修复 |
| 0-2 | Info | 记录在案 |

## 3D 严重程度公式

```
Score = R × 0.40 + I × 0.35 + C × 0.25
CVSS = Score / 3.0 × 10.0
```

### Reachability（可达性 R）

| 分数 | 描述 |
|------|------|
| 3 | 无需认证，公开可达 |
| 2 | 需要普通用户认证 |
| 1 | 需要管理员认证或内网访问 |
| 0 | 不可达（死代码/已禁用） |

### Impact（影响 I）

| 分数 | 描述 |
|------|------|
| 3 | RCE、完整系统控制、大规模数据泄露 |
| 2 | 敏感数据泄露、权限提升、完整数据读取 |
| 1 | 有限信息泄露、低权限操作 |
| 0 | 无实际影响 |

### Complexity（复杂度 C）

| 分数 | 描述 |
|------|------|
| 3 | 单请求可利用，无特殊条件 |
| 2 | 需要特殊 payload 或特定时机 |
| 1 | 需要多步链式攻击或竞态条件 |
| 0 | 有充分防护，不可利用 |

### 严重程度映射

| Score | CVSS | 等级 |
|-------|------|------|
| 2.70-3.00 | 9.0-10.0 | Critical |
| 2.10-2.69 | 7.0-8.9 | High |
| 1.20-2.09 | 4.0-6.9 | Medium |
| 0.10-1.19 | 0.1-3.9 | Low |

### 利用状态对分数的影响

| 状态 | 调整 |
|------|------|
| 已确认可利用 | 不调整 |
| 待验证 | C 减 1 |
| 环境依赖 | R 减 1，C 减 1 |
| 不可利用 | R=0, C=0 |

## 报告中的呈现

每个漏洞在报告中同时显示两种评分：

```
| DKTSS 优先级 | 8.0 (High) |
| 严重程度 | Critical (CVSS 9.3) |
| Reachability (R) | 3 - 无需认证，公开可达 |
| Impact (I) | 3 - RCE，完整系统控制 |
| Complexity (C) | 2 - 需要特殊 payload |
```

DKTSS 决定修复优先级，3D 严重程度决定影响范围。

## 攻击路径优先级评分

DKTSS 和 3D 评分决定"这个漏洞有多严重"，攻击路径评分决定"这个漏洞应该先修哪个"。两者结合才能给出正确的修复排序。

### 4 维度评分（总分 0-12）

| 维度 | 0 分 | 1 分 | 2 分 | 3 分 |
|------|------|------|------|------|
| **认证要求** | 公开可达 | 普通用户认证 | 管理员/特权认证 | 需多角色协作 |
| **请求复杂度** | 单请求，无前置条件 | 2-3 个请求，简单时序 | 多步链式攻击 | 需竞态条件/Timing |
| **社交工程依赖** | 无需交互，自动触发 | 需点击恶意链接 | 需登录并执行特定操作 | 需管理员主动操作 |
| **利用屏障** | 无防护，直接可达 | 有 WAF/防护但可绕过 | 需特定环境/版本 | 需内部网络/特殊配置 |

### 攻击路径等级

| 总分 | 等级 | 含义 |
|------|------|------|
| 10-12 | P0 | 极易利用，立即修复 |
| 7-9 | P1 | 较易利用，优先修复 |
| 4-6 | P2 | 需要一定条件，计划修复 |
| 0-3 | P3 | 利用困难，视情况修复 |

### 评分原则

- **认证要求**：端点是否需要认证，需要哪种级别的认证。公开可达 = 最高风险。
- **请求复杂度**：攻击需要多少个请求/步骤。单请求 = 最高风险。
- **社交工程依赖**：是否需要受害者配合。自动触发 = 最高风险。
- **利用屏障**：是否存在技术防护（WAF、版本限制、网络隔离）。无防护 = 最高风险。

### 与 DKTSS 的关系

攻击路径评分与 DKTSS 互补：
- DKTSS 回答："这个漏洞有多危险？"
- 攻击路径评分回答："攻击者利用这个漏洞有多容易？"
- 两个评分都高的漏洞 = 最优先修复

**示例**：
```
SQL 注入在后台管理页面：
- DKTSS: Base(7) - Friction(管理员-3) + Weapon(0) + Ver(0) = 4 (Low)
- 攻击路径: 认证(2) + 请求(0) + 社交(0) + 屏障(0) = 2 (P3)
→ 虽然 DKTSS 低，但因为需要管理员权限，攻击者实际难以利用

文件上传在公开 API：
- DKTSS: Base(9) - Friction(0) + Weapon(0) + Ver(0) = 9 (Critical)
- 攻击路径: 认证(0) + 请求(0) + 社交(0) + 屏障(0) = 0 (P0)
→ 两个评分都高，最优先修复
```

### 报告中的呈现

在漏洞列表中添加攻击路径列：

| 编号 | 标题 | 严重程度 | DKTSS | 攻击路径 | 状态 |
|------|------|----------|-------|----------|------|
| C-001 | ... | Critical | 9.2 | P0 (0/12) | CONFIRMED |
| H-001 | ... | High | 7.5 | P1 (7/12) | CONFIRMED |

每个漏洞详情中包含四维分解：

```
| 攻击路径优先级 | P0 (0/12) |
| - 认证要求 | 0 - 公开可达 |
| - 请求复杂度 | 0 - 单请求 |
| - 社交工程依赖 | 0 - 自动触发 |
| - 利用屏障 | 0 - 无防护 |
```
